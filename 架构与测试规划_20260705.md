# 屏幕视觉哨兵 - 架构与开发路线规划

## 1. 核心架构演进 (UI驱动 -> 数据驱动)
过去依靠 AHK 去模拟鼠标双击复制参数、判断界面红点颜色。这种方式极易受弹窗遮挡、软件卡顿和界面变化的影响，导致脚本崩溃（“UI驱动”）。
现在的目标是将项目升级为“数据驱动”：通过 Python 在后台调用 OCR 精准获取数字和文本，再由 AHK 接管这些纯净的数据进行业务逻辑判断。

## 2. 职责划分（双剑合璧）
*   **Python（视觉哨兵）—— 专门负责“看”**
    *   作为后台常驻服务（类似本地 API 接口）。
    *   负责根据预设坐标瞬间截图，并利用 RapidOCR 识别出数值（如 `QTC: 440`）和文本。
*   **AHK（值班助手）—— 专门负责“想”和“动”**
    *   通过 HTTP 请求向 Python 发起询问：“现在的参数是多少？”
    *   拿到 Python 返回的 JSON 数据后，执行医学逻辑判断。
    *   最后控制键盘鼠标进行自动化的纠错和填写操作。

## 3. 开发与测试路线图 (四步走)

### 第一步：实地踏勘（内网环境测试）
*   **行动**：带着打包好的 `ScreenVisionSentinel_Offline` 及其内置的 AHK 测试工作台前往科室。
*   **目标**：在医院真实系统上运行 Python 服务和 AHK 测试脚本。拾取“心室率”、“QTC”等参数的真实坐标，点击读取，验证 Python 能否准确传回 JSON 数据。就像是在系统里架设好各个参数的“摄像头”。
*   **产出**：固定的坐标数据；记录任何识别错误的案例（如图形截错、符号误认等）并带回真实截图。

### 第二步：优化“摄像头”（Python 端优化）
*   **行动**：带着科室的测试结果（截图）回家，利用项目中的 `mock.py` 在本地模拟科室环境。
*   **目标**：针对测试中发现的问题修改 Python 代码（比如针对 `|` 被认作 `1` 增加正则后处理纠错），并优化性能（提前裁剪区域防卡顿）。使后台 API 变得绝对可靠。

### 第三步：换掉“大脑”（AHK 业务逻辑重构）
*   **行动**：当 Python 接口稳定后，修改科室主力脚本 `Ginko操作优化...ahk`。
*   **目标**：植入“死规则”，接管医学标准的裁定权。
    *   *规则一（年龄与儿童标准）*：获取年龄，如果为空或大于 14 岁走成人心率标准；小于等于 14 岁走儿童标准。
    *   *规则二（QTC 裁判）*：自己设定男/女 QTC 标准上限。如果脚本利用获取的参数判断正常，但获取到的自动诊断包含“QT间期延长”，则 AHK 强行删除原诊断并替换为“正常心电图”。

### 第四步：终极闭环（全面上线）
*   带上优化好的 Python 服务端和全新的 AHK 脚本回到科室，彻底代替人工操作，完成业务闭环。

## 4. 解决离线环境开发痛点的方法
利用 `mock.py` 配合真实的科室全屏截图，在家中 100% 模拟医院内网环境进行 AI 辅助编程与测试，大幅减少在无环境的内网电脑上摸黑调 Bug 的摩擦感。
