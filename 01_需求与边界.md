# 需求与边界

最后更新日期：2026-07-05

## 项目背景

医院内部 Windows 软件常存在接口封闭、控件结构复杂、不可复制文本、远程桌面显示差异等问题。屏幕视觉哨兵采用截图和 OCR 方式读取屏幕像素，以类似人眼观察的方式读取屏幕上的文字、数字或状态。

项目已从早期“固定区域持续监控工具”演化为 Windows 本地通用屏幕视觉感知工具。

核心定位：

> 指哪里，看哪里，读取哪里。

## 目标用户

- 需要读取屏幕区域文字或数字的 Windows 用户。
- 在医院内部软件环境中需要本地、只读、可审计观察的工作人员。
- 需要通过 CLI、Server、AHK 或脚本复用屏幕 OCR 能力的开发者。

项目不得替代医生、技师或其他专业人员做临床判断。

## 三种读取模式

项目长期支持三类视觉读取方式：

- 点：用户或脚本指定屏幕点位，后续可按策略扩展为小区域读取。
- 矩形区域：用户或脚本指定 `left, top, width, height`，当前 CLI、Server 和 GUI 已围绕矩形区域工作。
- 持续观察范围：后续阶段对指定区域进行周期性读取、变化检测和记录。

当前实际可用能力主要是矩形区域的单次读取；持续观察尚未完成。

## 多入口设计

当前或计划入口：

- GUI：本地桌面交互、区域框选和手动测试。
- CLI：供 AHK、批处理或本地脚本调用。
- 本地 HTTP Server：避免重复加载 OCR 模型，服务 AHK 或其他本地程序。
- AHK：调用本地 Server 读取屏幕区域。
- 其他本地脚本：通过 CLI 或 Server 调用 Vision Core。

入口不得各自发明独立安全策略。CLI 和 Server 已使用 `VisionPipeline`；GUI 迁移是下一步任务。

## Vision Core 边界

Vision Core 只负责：

- 接收屏幕坐标或区域。
- 坐标与 DPI 换算。
- 区域合法性校验。
- 屏幕截图。
- 图像预处理（后续）。
- OCR。
- OCR 结果标准化。
- 调试截图安全保存。
- 性能和错误信息记录。
- 可选文字变化检测。

Vision Core 不得：

- 点击鼠标。
- 输入键盘。
- 提交报告。
- 修改患者信息。
- 作临床诊断。
- 代替医生进行决策。

## Automation Layer 边界

Automation Layer 未来才可能负责：

- 条件规则。
- 状态机。
- 提醒。
- 人工确认。
- 鼠标和键盘操作。
- AHK 联动。
- 紧急停止。
- 自动动作安全门控。

当前不进入自动化阶段。任何自动动作永久默认关闭。

## 当前功能需求

已实现：

- GUI 区域框选、手动坐标、单次截图和测试 OCR。
- CLI 单区域和多区域读取。
- Server 本地 OCR API。
- AHK helper 调用 Server。
- RapidOCR 和 Mock OCR。
- `VisionPipeline`、`CapturePolicy` 和安全调试截图策略。

后续阶段需求：

- GUI 迁移到 `VisionPipeline`。
- 固定间隔截图。
- 当前和上一轮识别结果展示完善。
- 内容变化判断和本地事件记录。
- 关键词和数字变化提醒。
- 连续多次一致确认。
- 冷却时间。
- 声音提醒和 Windows 通知。
- 人工确认后的受控动作。

## 非功能需求

- 默认只读。
- 本地优先。
- 不默认上传截图或 OCR 文本。
- UI 文本使用简体中文。
- 代码使用类型注解和 pathlib。
- 日志默认不得写入患者敏感信息。
- 异常不得静默吞掉。
- 配置可追踪、可复核。
- 后续长时间运行应具备稳定性和可恢复能力。

## 当前不做的功能

- 不做自动点击。
- 不做复杂规则引擎。
- 不做完整状态机。
- 不做临床相关自动决策。
- 不做自动审核或发布报告。
- 不做云端 OCR 默认方案。
- 不做 PyInstaller 打包。
- 不做 OCR 线程强制取消的伪实现。

## 医疗环境特殊要求

- 不得保存或提交真实患者姓名、住院号、身份证号、检查号。
- 未脱敏医院软件截图不得进入 Git。
- 测试数据必须使用虚构或脱敏内容。
- OCR 误识别必须作为常态风险处理。
- 数字变化必须经过确认策略，不能单次识别即触发高风险动作。
- 高风险、不可撤销或临床决策类动作不得自动化。

## 不依赖句柄的设计原则

项目不依赖：

- Windows 控件句柄。
- DOM。
- Accessibility API。
- 目标软件数据库。
- 目标软件插件或内部接口。
- 可复制文本。

这种设计降低目标软件内部结构变化带来的耦合，但会增加 OCR、DPI、窗口移动和截图定位风险。

## 隐私和数据最小化原则

- 默认不保存调试截图。
- 调试截图只能由布尔开关启用。
- 调试截图固定保存到 `data/debug/`。
- 调用方不得指定调试截图路径。
- 日志只记录请求 ID、成功状态、引擎、耗时、字符数和错误类别。
- 日志不得默认记录 OCR 全文、患者身份信息或调试截图路径。
- 本地配置、截图、数据库和日志默认不提交。

## MVP 定义

本项目 MVP 是一个本地只读视觉感知工具，能让用户或本地程序指定屏幕区域、截图、本地 OCR、显示或返回识别结果、判断变化并记录必要日志。MVP 不包含自动点击或临床相关自动操作。

## 项目边界

项目是视觉观察辅助工具，不是医院软件插件，不是 RPA 平台，不是诊断系统，不是报告审核系统。所有自动操作必须在后续阶段单独评估，并保持默认关闭。
