### 后台可视化与低负载优化

问题：

- 后台监测启动后只有手动状态弹窗，无法持续确认是否仍在运行、跑了几轮或最近一轮耗时，调试体验偏黑箱。
- 实测小空白区域在当前机器上使用整图文字检测平均约 `2830.6 ms`，而整块识别器模式平均约 `48.3 ms`；原有 2 秒轮询、空读相反模式重试和共享截图全空后的逐项重试会放大 CPU 占用与界面卡顿。

处理：

- AHK 工作台增加常驻后台状态栏，启动后每 2 秒查询轻量状态，显示运行/正在停止/已停止、心跳轮数、字段数、最近耗时、空值、复核和错误类型。
- 后台监测独立改为极速、单轮识别、默认 5 秒间隔；共享截图全空时等待下一轮，不再在同一轮逐项重复截图。普通手动 OCR 仍保留两轮稳定兜底。
- RapidOCR 的 ONNX Runtime CPU 线程限制为 intra-op 2、inter-op 1；昂贵的文字检测只跑一份规范化图像，小字段放大和二值化变体保留给较轻的整块识别器模式。
- 状态查询不再持有整轮 OCR 的监测锁；服务窗口每轮输出不含 OCR 原文和坐标的 heartbeat 日志。
- 根目录和离线包配置默认间隔改为 5 秒，离线包保留原有工作电脑坐标。

验证：

- `.venv\Scripts\python.exe -m pytest -o addopts='' -q`：通过，90 passed。
- `.venv\Scripts\python.exe -m ruff check src tests`：通过；Python 格式检查通过。
- 根目录与离线包 AHK 均通过 AutoHotkey v1 `--syntax-check`，并保持 UTF-8 BOM。
- 根目录与离线包启动脚本自检通过；`pip check` 返回 `No broken requirements found`。
- 本地 Mock HTTP 烟测完成两轮 heartbeat 后正常停止；状态含轮数、耗时、字段数和低负载标识，不含 OCR 原文、坐标或值指纹。
- 空白小区域本机复测：检测模式由优化前平均约 `2830.6 ms` 降至约 `1864.8 ms`；后台使用的整块识别器模式约 `48.4 ms/字段`。17 个内存空白字段的真实 RapidOCR 低负载整轮约 `920.1 ms`，且没有触发逐项重试。该数字只用于本机前后对比，不代表工作电脑实测结果。
- 仍需在工作电脑真实脱敏页面上观察 CPU 占用、心跳耗时和识别准确率，再决定是否把间隔继续提高到 8–10 秒。

### 后台最新参数接口与实时参数屏

问题：

- 常驻状态栏只能证明后台线程在运行，不能证明 OCR 参数已经通过接口传到 AHK 或按键精灵。
- 原 `/monitor/status` 按安全设计不包含 OCR 原文，因此不能直接承担参数传递。

处理：

- 新增 `GET /monitor/latest`，返回后台最新一轮 `texts` 和白名单字段结果；只在 Python 进程内保留一轮，重新启动监测时清空，服务退出后消失。
- `/monitor/status` 继续保持脱敏，不返回 OCR 值；数据接口不返回坐标、截图、识别框、请求 ID 或调试图路径，OCR 原文也不写日志。
- `/health` 新增 `monitor_latest_available: true`，API revision 更新为 `2026-07-10-monitor-latest-v2`，AHK 会识别旧服务并提示重新启动。
- AHK 工作台新增“实时参数屏”，每 2 秒读取一次 `/monitor/latest`，按“心电参数/患者信息”分组显示最新值、轮次、字段数、空值和复核数。
- 根目录与离线包的 Server、AHK 工作台同步更新，离线包原有工作电脑坐标保持不变。

接口意义：

- 实时参数屏能够显示并刷新参数，即证明后台 OCR → HTTP JSON → AHK 的数据链路已经打通。
- 按键精灵可用同一 GET 接口解析 `texts.HR`、`texts.PR` 等字段；后续无需重复截图即可消费最近一轮结果。
- 该接口只提供数据，不执行点击、输入、提交或医学判断；OCR 值仍须经过人工或既有复核逻辑确认。
