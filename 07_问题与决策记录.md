# 问题与决策记录

最后更新日期：2026-07-05

本文采用简化 ADR 格式：背景、决策、后果、状态。

## ADR-001：为什么不用 AHK 或按键精灵作为核心

背景：AHK 和按键精灵适合快速自动化鼠标键盘操作，但在医疗软件场景中存在误触、脚本散落、审计不足和默认动作风险。

决策：不以 AHK 或按键精灵作为项目核心。项目核心使用 Python 应用、明确接口、安全控制和测试。

后果：开发成本更高，但更容易审计、测试和分阶段控制风险。

状态：已确认。

## ADR-002：为什么采用视觉截图和 OCR

背景：目标软件可能没有可用控件句柄、DOM、Accessibility API 或可复制文本。

决策：采用截图和 OCR 作为主要观察路径，直接读取屏幕像素。

后果：减少对目标软件内部结构的依赖，但需要处理 OCR、DPI、窗口移动和分辨率变化风险。

状态：已确认。

## ADR-003：为什么 OCR 引擎必须可替换

背景：PaddleOCR、Tesseract、Windows OCR 在安装、准确率、速度和部署方面都有不确定性。

决策：建立 `BaseOCREngine`，业务逻辑只依赖统一接口。

后果：后续可以替换 OCR 引擎，但需要维护统一结果结构。

状态：已确认。

## ADR-004：为什么第一阶段只观察

背景：医疗相关界面存在高风险，识别错误或坐标错误可能导致严重后果。

决策：阶段 1 只观察、记录和显示，不提醒、不动作。

后果：功能推进较慢，但能先积累截图、OCR 和误识别样例。

状态：已确认。

## ADR-005：为什么动作执行与识别解耦

背景：如果识别逻辑直接触发鼠标或键盘动作，误识别会直接转化为操作风险。

决策：识别、检测、提醒、动作执行分层。动作必须经过安全控制和后续用户确认。

后果：代码路径更长，但可以插入安全开关、审计和紧急停止。

状态：已确认。

## ADR-006：为什么使用 PySide6

背景：项目需要 Windows 本地桌面界面、按钮状态、后续区域选择和长期运行 UI。

决策：使用 PySide6 作为桌面 UI。

后果：依赖体积较大，但 Windows 桌面体验较成熟。

状态：已确认。

## ADR-007：为什么使用 SQLite

背景：项目需要记录配置、事件和后续审计信息，但当前不需要服务器数据库。

决策：使用 SQLite 作为本地轻量存储。

后果：部署简单，适合个人内部工具；后续需要注意数据库文件隐私和备份。

状态：已确认。

## ADR-008：PaddleOCR 安装失败时的备用方向

背景：PaddleOCR 在 Windows 上可能因 Python 版本、PaddlePaddle 版本、模型下载或网络问题失败。

决策：当前不把 PaddleOCR 放入默认依赖。若安装失败，评估 Tesseract、Windows OCR 或离线模型缓存方案。

后果：阶段 0 不被大型依赖阻塞，但阶段 1 必须记录真实 OCR 实验结果。

状态：初步确认，待实验。

## ADR-009：是否保存原始截图

背景：原始截图可能包含患者信息和医院软件界面，存在隐私泄漏风险。

决策：默认不保存触发截图，`save_trigger_screenshots=False`。测试截图目录加入 `.gitignore`。

后果：隐私风险降低，但调试 OCR 时需要使用脱敏或虚构截图。

状态：初步确认，后续按实验需要复核。

## ADR-010：如何处理患者隐私

背景：项目当前场景涉及医院内部软件，患者姓名、住院号、身份证号、检查号和未脱敏截图都属于敏感内容。

决策：真实患者数据不得提交到 Git。测试数据必须脱敏或虚构。日志、截图、数据库默认本地忽略。

后果：开发和调试需要额外准备脱敏样例，但降低合规风险。

状态：已确认。

## ADR-011：项目定位演化

背景：项目早期定位为固定区域持续监控工具，但实际使用需求已经扩展为 GUI、CLI、Server、AHK 等多入口读取任意指定屏幕区域。

决策：项目定位调整为 Windows 本地通用屏幕视觉感知工具，核心思想是“指哪里，看哪里，读取哪里”。

后果：项目主干必须是统一 Vision Core，而不是某个 UI 或脚本入口。固定区域持续监控成为后续场景之一，不再是唯一架构中心。

状态：已确认。

## ADR-012：多入口设计

背景：GUI 适合手动框选和可视测试；CLI 适合批处理和脚本；本地 Server 适合避免 OCR 模型重复冷启动；AHK 适合接入既有 Windows 工作流。

决策：保留 GUI、CLI、Server、AHK 多入口，但入口只能负责参数接收和结果呈现，核心截图、OCR、调试图和安全校验应下沉到 Vision Core。

后果：入口数量增加，但公共逻辑不应复制。CLI 和 Server 已迁移到 `VisionPipeline`；GUI 迁移作为下一步任务。

状态：已确认。

## ADR-013：统一 VisionPipeline

背景：CLI 和 Server 曾分别解析区域、截图、保存调试图、调用 OCR 和拼 JSON，导致调试截图路径、区域限制和日志隐私策略分散。

决策：新增 `VisionPipeline` 统一完成区域校验、截图、OCR、可选安全调试图保存、脱敏日志和结构化结果返回。

后果：CLI 和 Server 共享同一视觉主干，减少重复逻辑。后续 GUI 迁移后可进一步统一行为。

状态：已确认。

## ADR-014：视觉安全与动作安全分离

背景：视觉读取风险包括区域过大、任意路径保存、OCR 明文日志和本地 Server 调用；动作安全风险包括点击、提交、审核、发布等高风险操作。

决策：使用 `CapturePolicy` 处理视觉输入和资源边界；保留 `SafetyController` 处理未来动作准入。两者不得混在一个类中。

后果：视觉层可以独立收口安全边界，同时不会暗示 OCR 结果可以直接触发动作。自动动作仍默认关闭。

状态：已确认。

## ADR-015：Server 串行或并发策略

背景：实际代码使用标准库 `HTTPServer`，不是 `ThreadingHTTPServer`。当前没有 RapidOCR 线程安全证据，也没有真正强制取消 OCR 的机制。

决策：本阶段保留串行 `HTTPServer`，只绑定 `127.0.0.1`。不引入 `ThreadPoolExecutor`，不虚假声明 OCR 可被超时取消。

后果：串行处理会让单个慢请求阻塞后续请求，但避免同一 RapidOCR 实例被并发调用。后续如需并发，必须先验证 RapidOCR 线程安全，或评估进程隔离。

状态：已确认。

## ADR-016：业务逻辑与识别引擎物理分离 (Python vs AHK)

背景：原计划中可能需要在 Python 端编写复杂的规则，或者在 AHK 中进行笨拙的模拟复制。医疗场景的判断规则（如男/女 QTC 标准、儿童心率标准）经常变动。

决策：Python 视觉哨兵仅作为本地后台 API（眼睛），绝对中立地输出 OCR 数字和文本 JSON。具体的医学死规则、逻辑冲突裁判（如参数未超标但原系统自动诊断异常时的强制改写）全部在 AHK/按键精灵客户端中实现。

后果：实现了“数据驱动”的降维打击。Python 服务可独立打包并保持高度稳定；医疗标准或纠错逻辑的变动只需修改轻量级的 AHK 脚本即可，彻底摆脱了系统 UI 的束缚。

状态：已确认，作为后续临床脚本改造的指导原则。
