# OCR 识别速度优化方案评估

## 背景与痛点

目前 `ahk_example.ahk` 中按 F5 一次性读取 6 个参数耗时极其漫长（可能需要将近 10 秒）。
这是因为每次调用 `SvsOCR()`，AHK 都在后台执行了一次 `svs-cli.exe`。
Python 脚本每次启动时，都需要重新加载庞大的 `onnxruntime` 库和 `RapidOCR` 模型，这个“冷启动”过程单次就需要约 1.5 秒。连续读取 6 个参数就是 6 次冷启动。

为了解决这个“非常慢”的问题，我提供两个方向的解决方案供你选择：

## 方案 A：批量读取改造（保守、简单）

**实现思路：**
修改 Python 的 CLI 工具，使其支持一次性接收多个坐标参数（比如：`--rects 1527,286,185,47|1897,292,142,41|...`）。AHK 端收集好所有的坐标，只调起 1 次 Python 程序，由 Python 一次性连续截 6 张图并识别，最后把 6 个结果组合成 JSON 返回给 AHK。

- **预期速度：** 读取 6 个参数的总耗时将从 9 秒骤降到 **1.5 秒 ~ 2 秒**。
- **优点：** 架构完全不用大改，AHK 脚本用完即走，不留后台进程，最稳定。
- **缺点：** 无论你读 1 个参数还是 6 个参数，都需要承受 Python 启动的那 **1.5 秒** 惩罚时间。

## 方案 B：常驻后台 API 服务（激进、极速）

**实现思路：**
将 Python 工具改造为一个轻量级的本地 HTTP 接口服务。
你只需要在工作前双击启动这个服务，它就会常驻后台并把模型加载到内存里“随时待命”。
AHK 脚本中不再通过 `WScript.Shell` 去运行 `exe`，而是通过 `WinHttpRequest` 瞬间向本地发送截屏指令。

- **预期速度：** 极速！读取单个参数耗时降至 **30~50 毫秒**。连读 6 个参数不超过 **0.5 秒**。
- **优点：** 真正的工业级速度，做到了“即指即读”，完全不会拖慢你 AHK 宏的运行节奏。
- **缺点：** 架构变重了。每次开工前，你必须确保这个 Python 后台服务正在运行，否则 AHK 获取不到数据。

## 用户审批需求

> [!IMPORTANT]
> 这两种方案的改造路线完全不同：
> - 如果你的 AHK 脚本里，1.5 秒的停顿是可以接受的，只是不能接受 10 秒，那么选 **方案 A**。
> - 如果你的脚本对时序要求极高，希望瞬间出结果，那么选 **方案 B**。
